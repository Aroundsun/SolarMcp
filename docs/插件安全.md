# 插件安全

本文总结 SolarMcp **当前已有的插件安全机制**、**已知缺口**与**建议实现顺序**，供后续迭代参考。

相关文档：[配置管理.md](./配置管理.md)、[插件ABI.md](./插件ABI.md)、[插件热重载.md](./插件热重载.md)。

---

## 1. 安全模型概览

当前采用 **「可信插件 + 网络 Token 门禁」** 模型：

- 谁能向 `plugins/` 写入 `.so`，谁就能被加载（与 `solarmcpd` **同进程、同用户**）
- 客户端须通过 Token 认证后才能调用 MCP 方法
- **没有**沙箱、代码签名、插件级 ACL

适用于**受信内网 / 开发环境**。不可信第三方插件场景需额外建设（见第 4 节）。

```
客户端 ──Token──► solarmcpd ──dlopen──► plugins/{name}/*.so
                      │                      │
                      │                      └─ 同进程原生代码，可调用任意 libc
                      └─ ABI 校验 + 试加载 + CallbackTool 边界
```

## 2. 当前已有防护

### 2.1 访问层（对外）

| 机制 | 说明 |
|------|------|
| Token 认证 | `auth.enabled: true` 时，须先 `authenticate`，否则拒绝 `tools/list`、`tools/call`、`plugins/reload` |
| 连接级状态 | 认证绑定 TCP 连接，非单次请求 |

配置见 `config.yaml` 的 `auth` 段。

**局限**：持有 token 的客户端可调用**所有**已注册工具；默认 token 须在生产环境更换。

### 2.2 加载层

| 机制 | 说明 |
|------|------|
| ABI 版本校验 | `mcp_plugin_abi_version()` 与 `SOLARMCP_PLUGIN_ABI` 不一致则拒绝加载 |
| 导出符号检查 | 缺少 `init` / `register` 等必需符号则加载失败 |
| 纯 C ABI | 插件不得跨 `.so` 使用宿主 C++ 对象；见 [插件ABI.md](./插件ABI.md) |
| 工具名去重 | 重复注册返回 `MCP_PLUGIN_ERR_DUP` |
| 热重载 validate | `reloadFromDirectory` / `reloadPlugin` 先试加载，成功后再替换 |
| 卸载顺序 | `unregisterTool` → `mcp_plugin_shutdown()` → `dlclose` |

**局限**：不校验 `.so` 签名或哈希；`plugins/` 目录写权限即等于可加载权限。

### 2.3 运行时边界

宿主通过 `mcp_host_api` 仅暴露：

- `register_tool`
- `log`
- `get_plugin_config_path`（插件自有 yaml 路径，Core 不解析内容）

工具执行经 `CallbackTool`：JSON 入参 → C 回调 → JSON 出参；宿主负责 `free` 插件分配的内存并校验输出 JSON。

**局限**：插件 `.so` 内仍可调用任意系统调用（读文件、联网等），无 seccomp / namespace 隔离。

### 2.4 工具层业务限制

#### 内置 `read_file`（`config.yaml` → `tools.read_file`）

- `realpath` 解析路径，缓解 `../` 目录穿越
- `allowed_paths` 前缀白名单
- `max_size_mb` 大小上限

#### 插件 `shell`（`plugins/shell/shell.yaml`）

- `enabled: false` 可不注册工具
- `timeout_sec` 执行超时
- `max_output_mb` 输出大小上限
- `allowed_shells` 解释器白名单（如 `/bin/sh`）

**局限**：`shell` 仍等价于**远程命令执行**；白名单只限制 shell 路径，不限制命令内容。

### 2.5 运维开关

| 配置 | 作用 |
|------|------|
| `plugins.autoload: false` | 启动时不自动加载插件 |
| `plugins.allow_reload: false` | 禁止 `plugins/reload` 热重载 |
| `plugins/shell/shell.yaml` → `enabled: false` | 禁用 shell 工具 |

## 3. 当前缺失

| 缺失项 | 风险 |
|--------|------|
| 插件代码签名 / 哈希校验 | 无法检测 `.so` 被篡改 |
| 插件目录 allowlist | 任意子目录下的 `.so` 均可被扫描加载（含 legacy `plugins/lib/` 残留） |
| 工具级 ACL | 认证通过后所有工具均可调用 |
| 沙箱 / 子进程隔离 | 恶意插件与主程序同权 |
| 资源配额（除 shell 超时/输出外） | 无 CPU / 内存上限 |
| 热重载并发保护 | 重载瞬间进行中的 `tools/call` 可能失败 |

---

## 4. 建议实现路线

与 [配置管理.md](./配置管理.md) 原则一致：**安全策略放 Core，业务参数放 Plugin**。

### P0 — 生产必做（成本低）

1. **插件目录治理**
   - 扫描时忽略 `plugins/lib/` 等 legacy 目录，或部署规范要求删除残留
   - Core 配置增加 `plugins.allowlist: [shell, example]`，只加载指定子目录

2. **默认配置硬化**
   - 强随机 `auth.token`；文档明确生产须更换
   - 生产模板：`allow_reload: false`；不需要 shell 时在 `shell.yaml` 设 `enabled: false`

### P1 — 信任链（中等工作量）

3. **插件完整性校验**
   - 构建生成 `.so.sha256` 或 `manifest.json`，加载前比对
   - 可选：Ed25519 / GPG 签名验证

4. **工具级 ACL**
   - 在 `tools/call` 前校验：token / 角色 → 允许的工具列表
   - 策略文件独立于插件 yaml（如 `config/policy.yaml`）

5. **热重载安全**
   - 重载与 `tools/call` 互斥或等待进行中调用结束
   - `plugins/reload` 可使用更高权限 token

### P2 — 强隔离（工作量大）

6. **插件沙箱 / 独立进程**
   - 高危工具（shell）放子进程，IPC 传 JSON
   - 或 seccomp-bpf / Landlock 限制 syscall、文件、网络

7. **资源配额**
   - 单工具执行时间、内存、输出全局上限

8. **最小权限运行**
   - 专用低权用户运行 `solarmcpd`；`plugins/` 只读

---

## 5. 推荐实现顺序

```
1. plugins.allowlist + 忽略 legacy lib/
2. .so 哈希校验（manifest 或 sidecar）
3. 工具级 ACL
4. 热重载并发锁
5. shell 子进程隔离（若仍需要 shell）
```

## 6. 生产环境检查清单

- [ ] `auth.enabled: true`，且 token 非默认值
- [ ] 不需要 shell 时：`plugins/shell/shell.yaml` → `enabled: false`
- [ ] 不需要热重载时：`plugins.allow_reload: false`
- [ ] 删除 `plugins/lib/` 等旧布局残留 `.so`
- [ ] `plugins/` 目录仅部署账号可写
- [ ] 服务不暴露公网，或前置防火墙 / VPN
- [ ] （后续）启用 allowlist 与完整性校验

## 8. 一句话结论

**现状是「可信插件 + Token 门禁 + 工具侧限流/白名单」，不是「不可信插件也可安全加载」。**  
上生产至少补：**目录/加载白名单 + 完整性校验 + 按需禁用或隔离 shell**；完整不可信插件场景再上沙箱或子进程。
